# Code Review — 2026-03-06

## Architecture Summary
`lunduke-transcripts` is a local-first CLI pipeline with three main entrypoints: `run` extracts transcript and frame artifacts from YouTube or local video into a canonical `tutorial_asset_bundle.json`, `tutorial` turns that bundle into a multi-agent written tutorial workflow, and `render` converts the final Markdown tutorial into HTML/PDF with screenshot validation. The main risk areas are the external process boundaries (`yt-dlp`, `ffmpeg`, Pandoc, Chrome-family PDF rendering, optional LLM routing), user-supplied file paths, and the tutorial pipeline’s state machine: it has to preserve clean handoff artifacts while coordinating multiple “editor” stages without letting one stage accidentally suppress the rest or leave stale public artifacts behind.

## Remediation Roadmap

### Fix Now (Blockers)
- R001 — As long as validation suppresses the other reviewers, the “co-editor” workflow is not actually happening.
- R002 — The product and code disagree on whether tutorial stages are advisory or blocking; this is the main behavior the user asked to change.
- R003 — The reroute state machine has a latent crash path that will surface once the advisory flow drives more writer/visual-only revisions.

### Fix Soon (High ROI)
- Update tutorial manifests and docs so “latest draft/final artifact” semantics are explicit and no stale final PDF is mistaken for a fresh run.

### Fix Later (Refactors)
- Consolidate tutorial status handling so `awaiting_outline_approval`, editorial warnings, and fatal runtime failures are represented distinctly without overloading `publish_eligible`.

## Patch Suggestions

For R001, remove the validation gate that replaces real review artifacts with skipped stubs.

